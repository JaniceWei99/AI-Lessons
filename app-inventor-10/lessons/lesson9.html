9.2 开发工作流9.2 Development Workflow

专业开发者不会一口气把整个 App 写完。他们遵循一个循环流程，每次只做一小步，确保每一步都能正常工作后再继续。这就像搭积木——一块一块稳稳地搭上去，而不是一次性堆一大堆然后祈祷它不会倒。 Professional developers don't build an entire App in one go. They follow a cyclical workflow, taking one small step at a time and making sure each step works before moving on. It's like building with LEGO — you place one brick at a time steadily, instead of stacking a huge pile and hoping it doesn't collapse.

1

计划Plan

决定下一个要实现的功能是什么Decide what feature to build next

2

3

测试Test

在手机上运行，看看是否正常Run it on your phone to see if it works

4

修复Fix

5

重复Repeat

回到第 1 步，开始下一个功能Go back to step 1, start the next feature

黄金法则：每次只做一件事！

不要同时添加 3 个功能。先完成一个功能，测试通过后再开始下一个。这样当出现 Bug 时，你就能很快知道是哪个新功能引起的问题。

举例：
1. 先做"拍照"功能 → 测试 → 通过
2. 再做"调用 AI 识别" → 测试 → 通过
3. 最后做"显示结果" → 测试 → 通过

如果你一口气全做完再测试，发现不工作了，你就不知道是拍照有问题、API 调用有问题、还是显示有问题了。 Golden Rule: Do ONE thing at a time!

Don't add 3 features simultaneously. Complete one feature, test it, and only then start the next. This way, when a bug appears, you'll quickly know which new feature caused it.

Example:
1. Build the "take photo" feature → test → pass
2. Then build "call AI recognition" → test → pass
3. Finally build "display results" → test → pass

If you build everything at once and then test, and it doesn't work, you won't know if the problem is with the camera, the API call, or the display.

在开发过程中，建议你在纸上或笔记本上列一个功能清单（Feature List），每完成一个就打个勾。这样你可以清楚地看到进度，也不会忘记要做什么。 During development, we recommend keeping a feature checklist on paper or in a notebook. Check off each feature as you complete it. This way you can clearly see your progress and won't forget anything.

9.3 常见错误与调试9.3 Common Bugs & Debugging

Bug（程序错误）是开发过程中完全正常的事情——即使是最厉害的程序员也会遇到 Bug！关键是学会如何快速找到并修复它们。下面是使用 App Inventor + AI API 时最常见的 5 种错误，以及解决方法： Bugs are a completely normal part of development — even the best programmers encounter them! The key is learning how to quickly find and fix them. Here are the 5 most common errors when using App Inventor + AI APIs, along with solutions:

Bug #1：API 调用返回错误Bug #1: API Call Returns Error

现象：你的 App 调用 API 后没有收到正确的结果，而是得到了一个错误信息，比如 Error 1101 或 responseCode: 401。

常见原因：
- API 的 URL 地址写错了（多了一个字母或少了一个斜杠）
- 请求头（Headers）缺少了必要的信息，比如 Content-Type 或 Authorization
- 请求方法用错了（应该用 POST 却用了 GET）

解决方法：
1. 仔细对比你的 URL 和 API 文档中的 URL，一个字符一个字符地检查
2. 确保 Headers 中包含 Content-Type: application/json 和正确的 Authorization: Bearer sk-xxx...
3. 检查你使用的请求方法（GET/POST）是否和 API 文档一致 Symptom: Your App calls the API but doesn't receive the correct result. Instead, you get an error message like Error 1101 or responseCode: 401.

Common Causes:
- The API URL is wrong (an extra letter or a missing slash)
- Required headers are missing, like Content-Type or Authorization
- Wrong request method (using GET instead of POST)

Solution:
1. Carefully compare your URL with the API documentation character by character
2. Make sure Headers include Content-Type: application/json and the correct Authorization: Bearer sk-xxx...
3. Check that your request method (GET/POST) matches the API documentation

Bug #2：JSON 解析失败Bug #2: JSON Parsing Fails

现象：API 返回了数据，但你的 App 无法正确读取其中的内容，显示空白或者报错。

常见原因：
- JSON 路径（Path）写错了。比如数据在 choices[0].message.content，但你写成了 choices.message.content（忘记了 [0]）
- API 返回的 JSON 结构和你预期的不一样

解决方法：
1. 先用 Notifier 把 API 返回的原始数据完整显示出来，看看 JSON 的真实结构
2. 使用 select list item 积木块正确地访问 JSON 中的数组元素
3. 逐层解析——先取最外层，再取第二层，一步一步深入 Symptom: The API returns data, but your App can't read the content correctly — it shows blank or throws an error.

Common Causes:
- The JSON path is wrong. For example, the data is at choices[0].message.content, but you wrote choices.message.content (forgot [0])
- The API's actual JSON structure is different from what you expected

Solution:
1. First use a Notifier to display the raw API response so you can see the actual JSON structure
2. Use the select list item block to correctly access array elements in the JSON
3. Parse layer by layer — get the outermost layer first, then the second layer, step by step

Bug #3：App 在手机上闪退Bug #3: App Crashes on Phone

现象：App 在手机上打开后突然关闭，或者某个功能执行时闪退。

常见原因：
- 缺少必要的权限（比如使用相机但没有请求相机权限）
- 尝试访问一个不存在的组件或变量
- 内存不足（比如加载了一张特别大的图片）

解决方法：
1. 在 Designer 视图中检查你是否添加了所有需要的组件（Camera、Web 等）
2. 确保在 Screen1 的属性中设置了正确的权限
3. 在使用变量前，先确认变量已经被初始化（赋予了初始值）
4. 图片大小尽量控制在 1MB 以内 Symptom: The App suddenly closes after opening on your phone, or crashes when executing a certain function.

Common Causes:
- Missing required permissions (e.g., using the camera without requesting camera permission)
- Trying to access a component or variable that doesn't exist
- Out of memory (e.g., loading a very large image)

Solution:
1. In the Designer view, check that you've added all required components (Camera, Web, etc.)
2. Make sure you've set the correct permissions in Screen1's properties
3. Before using a variable, make sure it has been initialized (given a starting value)
4. Keep image sizes under 1MB whenever possible

Bug #4：API Key 过期或被限流Bug #4: API Key Expired or Rate Limited

现象：你的 App 之前一直能正常调用 API，但突然不工作了。API 返回 429 Too Many Requests 或 401 Unauthorized。

常见原因：
- API Key 已经过期或被禁用了
- 你在短时间内调用 API 次数太多，触发了限流
- 免费额度已经用完

解决方法：
1. 登录 API 提供商的网站，检查你的 API Key 是否仍然有效
2. 查看你的用量统计，看看是否超出了限制
3. 如果是限流，等几分钟后再试
4. 在 App 中添加错误处理——当收到错误时显示友好的提示信息，而不是直接崩溃 Symptom: Your App used to call the API just fine, but suddenly stopped working. The API returns 429 Too Many Requests or 401 Unauthorized.

Common Causes:
- The API Key has expired or been disabled
- You made too many API calls in a short period, triggering rate limiting
- Your free quota has been used up

Solution:
1. Log into the API provider's website and check if your API Key is still valid
2. Check your usage statistics to see if you've exceeded the limit
3. If it's rate limiting, wait a few minutes and try again
4. Add error handling in your App — show a friendly message to the user when an error occurs instead of crashing

Bug #5：图片太大，API 无法处理Bug #5: Image Too Large for API

现象：你的 App 发送图片给 AI API 进行识别，但 API 返回错误或者超时。

常见原因：
- 手机拍的照片通常有 3-10MB，很多 API 有文件大小限制
- 图片转换成 Base64 编码后会变得更大（约增加 33%）
- 网络速度慢时，大文件传输容易超时

解决方法：
1. 在发送前压缩图片——使用 App Inventor 的 Canvas 组件来缩小图片尺寸
2. 将图片的宽度和高度限制在 1024 像素以内
3. 如果 API 支持 URL 方式传图，可以先把图片上传到云存储，再把 URL 传给 API
4. 增加 Web 组件的超时时间设置 Symptom: Your App sends an image to the AI API for recognition, but the API returns an error or times out.

Common Causes:
- Phone photos are usually 3-10MB, and many APIs have file size limits
- Converting images to Base64 encoding makes them even larger (~33% increase)
- Large files easily time out on slow networks

Solution:
1. Compress the image before sending — use App Inventor's Canvas component to resize the image
2. Limit image width and height to 1024 pixels
3. If the API supports URL-based images, upload the image to cloud storage first, then send the URL to the API
4. Increase the Web component's timeout setting

9.4 调试技巧9.4 Debugging Tips

找 Bug 就像当侦探一样——你需要收集线索、分析证据、缩小范围，最终找到"罪犯"。这里有 5 个超实用的调试技巧： Finding bugs is like being a detective — you need to collect clues, analyze evidence, narrow down suspects, and ultimately catch the "culprit." Here are 5 super practical debugging tips:

技巧 1：使用 "Do it" 功能Tip 1: Use the "Do It" Feature

在 App Inventor 的 Blocks 编辑器中，当你的手机已经通过 AI Companion 连接时，你可以右键点击任何一个积木块，然后选择 "Do it"。这会立即执行这个积木块，并在旁边显示结果。

这是一个非常强大的功能！比如，你可以右键点击 get global variable 积木块来查看变量当前的值，或者右键点击一个数学运算积木块来看看计算结果是否正确。

使用场景：当你不确定某个积木块是否产出了正确的值时，"Do it" 可以让你立即看到答案。 In App Inventor's Blocks editor, when your phone is connected via AI Companion, you can right-click any block and select "Do It." This immediately executes that block and shows the result right next to it.

This is an incredibly powerful feature! For example, you can right-click a get global variable block to see the variable's current value, or right-click a math block to check if the calculation is correct.

When to use it: Whenever you're unsure if a block is producing the correct value, "Do It" lets you see the answer immediately.

技巧 2：添加 Notifier 显示调试信息Tip 2: Add Notifier for Debug Messages

在 Designer 视图中，从 User Interface 面板拖入一个 Notifier 组件。然后在 Blocks 视图中，你可以在程序的关键位置添加 call Notifier1.ShowAlert 积木块来显示调试信息。

举例：
- 在调用 API 之前，显示请求数据：Notifier1.ShowAlert("发送的数据: " + requestBody)
- 在收到 API 回复后，显示原始返回内容：Notifier1.ShowAlert("收到: " + responseContent)
- 在发生错误时，显示错误详情：Notifier1.ShowAlert("错误: " + errorMessage)

记住：调试完成后，记得删除或禁用这些调试用的 Notifier 积木块，否则用户在使用你的 App 时会看到这些技术信息。 In the Designer view, drag a Notifier component from the User Interface palette. Then in the Blocks view, add call Notifier1.ShowAlert blocks at key points in your program to display debug information.

Examples:
- Before calling the API, show the request data: Notifier1.ShowAlert("Sending: " + requestBody)
- After receiving the API response, show the raw content: Notifier1.ShowAlert("Received: " + responseContent)
- When an error occurs, show error details: Notifier1.ShowAlert("Error: " + errorMessage)

Remember: After debugging, remove or disable these debug Notifier blocks, or your users will see technical messages when using your App.

技巧 3：先用小数据测试Tip 3: Test with Small Inputs First

在测试 AI 功能时，不要一上来就用很复杂的输入。先用最简单、最短的输入来测试，确认基本功能正常后再逐步增加复杂度。

举例：
- 测试 AI 聊天：先发送 "Hi"，看看能不能收到回复
- 测试图片识别：先用一张很小的、很清晰的图片
- 测试语音识别：先说一个简单的单词，比如 "Hello"

如果简单的输入就出错了，那问题一定在基础设置上（URL、Key、Headers），不需要往复杂的地方想。 When testing AI features, don't start with complex inputs. Start with the simplest, shortest inputs first, confirm basic functionality works, and then gradually increase complexity.

Examples:
- Testing AI chat: Send "Hi" first, see if you get a reply
- Testing image recognition: Start with a small, clear image
- Testing speech recognition: Say a simple word like "Hello" first

If simple inputs already cause errors, the problem is definitely in the basic setup (URL, Key, Headers) — no need to look for complex issues.

技巧 4：在浏览器中单独测试 APITip 4: Test API Separately in a Browser

如果你不确定是 App 的问题还是 API 的问题，可以先在浏览器中单独测试 API。使用像 Hoppscotch（hoppscotch.io）这样的在线工具，手动输入 URL、Headers 和请求体来测试 API。

如果在浏览器中 API 能正常返回结果，那说明问题在你的 App Inventor 积木块上。如果浏览器中也返回错误，那说明 API 本身有问题（URL 不对、Key 无效等）。

这种方法叫做"隔离测试"——通过分别测试不同的部分，来缩小问题的范围。 If you're not sure whether the problem is with your App or the API, test the API separately in a browser. Use an online tool like Hoppscotch (hoppscotch.io) to manually enter the URL, Headers, and request body to test the API.

If the API returns correct results in the browser, then the problem is in your App Inventor blocks. If the browser also returns an error, then the API itself has an issue (wrong URL, invalid Key, etc.).

This method is called "isolation testing" — by testing different parts separately, you narrow down where the problem is.

技巧 5：认真阅读错误信息Tip 5: Read Error Messages Carefully

很多同学看到错误信息就慌了，直接跳过不看。但其实错误信息就是答案的线索！养成仔细阅读错误信息的习惯。

常见错误代码的含义：
- 400 Bad Request = 你发送的数据格式不对
- 401 Unauthorized = API Key 错误或缺失
- 403 Forbidden = 没有权限访问这个 API
- 404 Not Found = URL 地址写错了
- 429 Too Many Requests = 调用太频繁，被限流了
- 500 Internal Server Error = API 服务器出了问题（不是你的错！）

把错误信息看作你的朋友，它在告诉你哪里出了问题。 Many students panic when they see error messages and skip right past them. But error messages are clues to the answer! Make a habit of reading error messages carefully.

Common error code meanings:
- 400 Bad Request = Your request data format is wrong
- 401 Unauthorized = API Key is wrong or missing
- 403 Forbidden = No permission to access this API
- 404 Not Found = URL address is wrong
- 429 Too Many Requests = Too many calls, rate limited
- 500 Internal Server Error = API server has a problem (not your fault!)

Think of error messages as your friend — they're telling you what went wrong.

9.5 UI 打磨9.5 UI Polish

一个优秀的 App 不仅功能要好用，界面也要看起来舒服、专业。在你的 App 基本功能都完成后，花一些时间来打磨 UI（用户界面），让你的 App 从"能用"变成"好用又好看"。 A great App isn't just functional — it should also look polished and professional. After your App's core features are working, spend some time polishing the UI (User Interface) to transform your App from "works" to "works beautifully."

统一颜色和间距Consistent Colors & Spacing

选择 2-3 种主要颜色，在整个 App 中统一使用。比如按钮都用蓝色、标题都用深灰色、背景用浅色。组件之间留出统一的间距，不要有的地方挤在一起，有的地方空一大片。

小技巧：在 App Inventor 中，你可以使用 HorizontalArrangement 和 VerticalArrangement 来精确控制布局和间距。 Choose 2-3 main colors and use them consistently throughout your App. For example, buttons in blue, titles in dark gray, backgrounds in a light color. Keep consistent spacing between components — don't have some areas crammed together and others with huge gaps.

Pro tip: In App Inventor, use HorizontalArrangement and VerticalArrangement to precisely control layout and spacing.

加载指示器Loading Indicators

当你的 App 正在等待 API 回复时，用户可能会以为 App 卡住了。添加一个加载提示来告诉用户"正在处理中"。

实现方法：
1. 添加一个 Label，文字设为"正在思考中..."
2. 默认设置 Visible = false（不可见）
3. 在发送 API 请求前，设置 Visible = true（显示）
4. 在收到 API 回复后，设置 Visible = false（隐藏） When your App is waiting for an API response, users might think the App is frozen. Add a loading indicator to tell users "processing...".

How to implement:
1. Add a Label with text "Thinking..."
2. Set Visible = false (hidden) by default
3. Before sending the API request, set Visible = true (show)
4. After receiving the API response, set Visible = false (hide)

用户友好的错误提示User-Friendly Error Messages

当 API 调用失败时，不要给用户显示 Error 401 Unauthorized 这样的技术信息。用通俗易懂的语言告诉用户发生了什么，以及他们可以做什么。

好的错误提示示例：
- "网络连接似乎有问题，请检查 Wi-Fi 后重试。"
- "AI 小助手暂时忙不过来了，请稍后再试。"
- "请拍一张更清晰的照片，这样 AI 才能更好地识别。" When an API call fails, don't show users technical messages like Error 401 Unauthorized. Use plain language to tell them what happened and what they can do.

Good error message examples:
- "There seems to be a network issue. Please check your Wi-Fi and try again."
- "The AI assistant is a bit busy right now. Please try again in a moment."
- "Please take a clearer photo so the AI can recognize it better."

- 选择和你的 App 颜色一致的配色

Recommended tools:

If you still have many features left to build, focus on the most essential feature. An App with one feature done well is far better than an App with ten half-finished features.

If your core features are already working, spend your time polishing the UI, adding loading states, and handling errors. These seemingly small details can dramatically improve your App's quality!

9.7 挑战任务9.7 Challenge

课堂测验Lesson Quiz

1. 专业开发者推荐的开发流程是什么？1. What is the development workflow recommended by professional developers?

专业开发者遵循"计划 → 搭建 → 测试 → 修复 → 重复"的循环流程，每次只完成一个小功能，确保它正常工作后再继续下一个。Professional developers follow the "Plan → Build → Test → Fix → Repeat" cycle, completing one small feature at a time and making sure it works before moving on.

2. 当你的 API 调用返回 401 Unauthorized 错误时，最可能的原因是什么？2. When your API call returns a 401 Unauthorized error, what is the most likely cause?

401 Unauthorized 表示认证失败。最常见的原因是 API Key 写错了、过期了或者在请求头中缺少 Authorization 字段。401 Unauthorized means authentication failed. The most common cause is that the API Key is wrong, expired, or the Authorization header is missing from the request.

3. App Inventor 中的 "Do it" 功能有什么作用？3. What does the "Do It" feature in App Inventor do?

"Do it" 功能可以在手机连接 AI Companion 时，右键点击任何积木块来立即执行它，并在旁边显示结果。这对于检查变量值和调试非常有用。The "Do It" feature lets you right-click any block while connected to AI Companion to immediately execute it and display the result. This is very useful for checking variable values and debugging.

4. 当你的 App 等待 API 回复时，应该怎么做来提升用户体验？4. What should you do to improve user experience while your App waits for an API response?

当 App 等待 API 回复时，应该显示加载提示（如"正在思考中..."），让用户知道 App 正在工作，而不是卡住了。这是良好用户体验的基本做法。When waiting for an API response, show a loading indicator (like "Thinking...") so users know the App is working, not frozen. This is a fundamental practice for good user experience.

5. "隔离测试"是什么意思？5. What does "isolation testing" mean?

"隔离测试"是指分别测试程序的不同部分（比如先在浏览器中单独测试 API，再测试 App 的积木块），通过排除法来确定问题出在哪里。"Isolation testing" means testing different parts of your program separately (e.g., testing the API in a browser first, then testing your App blocks), using the process of elimination to determine where the problem lies.
